# Data Sharing, Repositories, and Platforms

## Purpose

Describe what the current sources support about sharing pathogen genomics data and choosing repositories or platforms.

## Guidance Draft

Pathogen genomics data sharing has two different purposes that need to be kept separate. One is operational sharing, where data, reports, histories, or workspaces are exchanged between laboratories, public-health teams, collaborators, and services. The other is public or controlled release into repositories and surveillance platforms so that data can support wider outbreak detection, research, method development, and international comparison.

The PHA4GE infrastructure source treats sharing as part of data flow. Its vignettes show several implementation patterns: data kept local to a laptop, sharing mediated by HPC administrators, cloud object storage, platform workspaces, Galaxy histories, and IRIDA projects. These examples are useful because they show that sharing is not a single function. It depends on the operating model, platform, identity system, data sensitivity, and intended user.

Armstrong et al. provide the strongest public-health argument for openness. They describe US public-health agencies uploading sequence data on foodborne bacterial pathogens, influenza, and other pathogens to public databases hosted by NCBI, as well as contributions to ReSeqTB and GISAID. They also state the important qualification: openness cannot be complete and unconditional because public-health agencies must consider confidentiality and risk.

The PHE case study provides a concrete repository example. It reports that Salmonella sequences generated by the reference laboratory were uploaded to the Sequence Read Archive through NCBI. This is useful evidence that repository deposition can be part of routine public-health service design, not only research publication.

A useful guidance chapter should therefore avoid a simplistic "share everything" position. Programmes need to decide what data should be shared, with whom, at what stage, through which repository or platform, under what access model, and with what metadata. They also need to decide which data are public, which are shared through trusted collaborations, and which remain restricted because of confidentiality, legal, ethical, or operational risks.

The dynamic guide now has a beta decision pathway for data-sharing and repository routing. It separates purpose, route, data type, minimum metadata, sensitive metadata, quality status, access model, identifiers and accessions, terms of use, attribution, benefit-sharing expectations, correction, withdrawal, and accountable ownership. It also has a governance-review checklist for local review before data leave the programme or move into a shared platform.

This is still not a repository-specific submission manual. The current sources support the decision structure, the need for governed sharing, access models, provenance, identifier tracking, and public versus restricted metadata. Exact INSDC/NCBI, GISAID, national-platform, or organism-specific submission fields still need direct extraction from current platform documentation and local policy sources before they can be prescribed.
